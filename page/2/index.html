Flume之Sink Processors以及如何调优

2019-09-22
Flume

Sink Processors

当你的Agent有多个Sink时，Flume提供两种策略：failover和load_balance，下面详细介绍一下两种策略。

load_balance

它作用是保持event到下一个agent的负载均衡，内置两种策略：round_robin(轮询)和random(随机)，默认的策略是round_robin。如果配置的某个sink挂掉，它尝试将evnet向其他sink发送，如果配置的sink全挂掉，那么就会报错。
示例配置：

failover

它的作用是保证event会被一个agent接受，也就是容错，它会保持一个优先级的sink队列，在其中数字越大优先级越高，当你最高优先级的sink对应的agent挂了，会将evnet转移到优先级第二高的sink。
示例配置：

调优

如何对flume进行调优呢？

主要是围绕三个核心组件的选型来调优

我们在初始Flume中就介绍过三个组件常用的类型。

Source

因为常用的也就是taildir类型，默认的batchSize参数为100过小，它表示一次最大读取到channel的行数，可以根据生产实际数据量来进行调整。

Channel

参数capacity代表最大在channel里存储的evnet数量，参数transactionCapacity代表source或者sink在channel里操作的最大事务数。

值得注意的是：设置的batchSize参数要小于等于参数transactionCapacity

Sink

sink的调参参数也有batchSize，方法也是跟source一样。

但是sink生产上用得就是hdfs类型，对于此也有三个参数需要调整：

  1. hdfs.rollInterval

    代表等待多少秒hdfs文件滚动，设置为0代表不使用，默认是30秒
  2. hdfs.rollSize

    代表收集到多少字节的文件滚动，设置为0代表不使用，默认是1024字节
  3. hdfs.rollCount

    代表收到多少数量的event滚动，设置为0代表不使用，默认是10

以上三个参数达到任意一条，文件都会滚动。

如果收集到hdfs上需要压缩的话，则要改动hdfs.fileType和hdfs.codeC 两个参数。

容错

在上面提及的failover，也是调优的一部分，提高整个flume工作链的健壮。

初识Flume

2019-09-18
Flume

Flume是什么

Apache Flume是一个分布式，可靠且可用的系统，用于有效地从许多不同的源收集，聚合和移动大量日志数据到集中式数据存储(例如HDFS)。

Apache Flume的使用不仅限于日志数据聚合。 由于数据源可以是自定义的，因此Flume可用于传输大量事件数据，包括但不限于网络流量数据，社交媒体生成的数据，电子邮件消息以及几乎任何可能的数据源。

Flume有两条产品线，0.9.x版本和1.x版本，但是0.9.x版本没有人使用，官方也推荐使用1.x版本的。

核心概念

Flume中最小单元是Agent，日志收集可以单Agent完成，也可以多个Agent串行或者并行完成，下图则是个最简单的单Agent架构图。

Source

  • Source主要是收集数据
  • 常用的Source类型也就是avro(Agent和Agent传输的类型)/taildir/netcat
  • Source至少与一个Channel相连

Channel

  • Channel起到缓冲通道的作用
  • 常用的Channel类型有：memory/file，memory与file区别就在于数据是否持久化，选择memory可能在宕机的时候存在Channel数据丢失，选择file可能会对你的硬盘容量是个挑战
  • 事务保证sink数据到Channel和Channel数据到Sink
  • Channel可以与任意数量的Source和Sink搭配

Sink

  • Sink将采集的日志数据发送给HDFS或者下一个Agent
  • 常用的Sink类型有：avro/hdfs/logger(测试使用，打印在控制台)
  • Sink必须至少有个Channel相连

Event

  • 整个Agent中数据传输的基本单元
  • 由Map<String, String>类型的Headers和byte[]数组的body组成，其中Headers是描述存放在body数据可以为空，body是真正存放数据

额外练习

需求：监控一个文件夹下的数据输出到HDFS并且分区

需求分析

三大组件类型选择：

Source：taildir

Channel：memory

Sink：hdfs

taildir/exec/spooling区别

exec：只能监控一个文件，在重新启动之后容易造成数据重复

spooling：可以监控文件夹，但是重新启动之后容易造成数据丢失

taildir：可以监控多个文件，会在采集过程记录采集位置，重新启动之后不会丢失数据或者数据重复
